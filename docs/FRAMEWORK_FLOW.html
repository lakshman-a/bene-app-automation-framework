Playwright Test Framework — God Book

Your one-page guide for writing, running, and reading a UI test in playwright-java-framework-v3-main. Bookmark this file.

1. How the engine actually works (read this once)

When you type mvn test, this is the chain of things that fire — like a relay race:

Maven Surefirefinds TestRunner
TestRunnerloads features & glue
Hooks @Beforeopens browser
Feature fileGherkin scenario
Step Defsmatch Gherkin → Java
Page Objectclicks, types, reads
Hooks @Afterscreenshot on fail, close browser
Alluregenerates report

Bene-style analogy: Think of it like processing a withdrawal request. Surefire is the system entry. The TestRunner is the queue picker. Hooks open and close the user session. The feature file is the request form (what the customer asks for in English). Step defs translate the form into actual API calls. The Page Object is the screen / endpoint that does the work. Allure is the audit log.

The seven living parts of your framework

PartWhere it livesWhat it does
Feature file (Gherkin)src/test/resources/features/*.featurePlain-English scenario: Given / When / Then
Step Definitionsrc/test/java/com/upskill/stepdefinitions/Java code that runs when Cucumber sees a Gherkin line
Page Object (POM)src/main/java/com/upskill/pages/<app>/One class per screen. Holds locators + actions for that screen
BasePagesrc/main/java/com/upskill/pages/BasePage.javaParent of every Page Object. Provides click(), type(), waitForElement(), etc.
Driversrc/main/java/com/upskill/driver/PlaywrightDriver.javaOpens/closes the browser. One per thread (parallel-safe).

2. To add a NEW UI test — do these 7 steps in this order

Example: "I want to test that a beneficiary login on SauceDemo (a stand-in for bene portal) shows the Products page."

1

Describe the scenario in plain English

Open the feature file for your app (or create a new .feature in src/test/resources/features/). Use Given / When / Then:

@all @bene
Feature: Beneficiary login
  Background:
    Given I am on the SauceDemo login page

  @smoke
  Scenario: Standard beneficiary lands on Products page
    When I login with username "standard_user" and password "secret_sauce"
    Then I should be redirected to the Products page

@smoke@regression @all are tags — like labels. You filter at run-time with -Dcucumber.filter.tags="@smoke".

2

Find the elements on the page

Open the app in Chrome → right-click element → Inspect. Grab one of:

  • #user-name (id) — best
  • [data-test='error'] (data-attribute) — also great
  • .error-message (CSS class) — OK
  • XPath — last resort, brittle
Tip: Playwright lets you record locators too. Run npx playwright codegen https://www.saucedemo.com in a terminal and it opens a browser that emits selectors as you click.
3

Put those locators + actions into a Page Object

One class per screen. Lives in src/main/java/com/upskill/pages/<app>/. Always extends BasePage:

public class LoginPage extends BasePage {
    // ===== Locators (constants at top) =====
    private static final String USERNAME_FIELD = "#user-name";
    private static final String PASSWORD_FIELD = "#password";
    private static final String LOGIN_BUTTON   = "#login-button";

    // ===== Actions (one method per user intent) =====
    @Step("Login as: {username}")
    public void loginAs(String username, String password) {
        type(USERNAME_FIELD, username);
        type(PASSWORD_FIELD, password);
        click(LOGIN_BUTTON);
    }

    // ===== Validations (one method per question) =====
    public boolean isErrorDisplayed() {
        return isVisible("[data-test='error']");
    }
}
Why the structure matters: Locators at the top → easy to update when the UI changes. Actions in the middle → describe what a user does. Validations at the bottom → answer yes/no questions for the test. Never write page.locator(...) in a step definition. That belongs in the Page Object.
4

Wire the Gherkin steps to your Page Object methods

In src/test/java/com/upskill/stepdefinitions/ create or edit a Step Def class:

public class LoginSteps {
    private LoginPage loginPage = new LoginPage();
    private ProductsPage productsPage = new ProductsPage();

    @Given("I am on the SauceDemo login page")
    public void iAmOnLoginPage() { loginPage.navigateToLoginPage(); }

    @When("I login with username {string} and password {string}")
    public void iLogin(String user, String pwd) { loginPage.loginAs(user, pwd); }

    @Then("I should be redirected to the Products page")
    public void iSeeProducts() {
        assertTrue("Products page expected", productsPage.isProductsPageDisplayed());
    }
}

Each @Given/@When/@Then string must match the Gherkin line word-for-word (parameters in {string} / {int}).

5

Run it

From the project root, in PowerShell:

mvn test "-Dcucumber.filter.tags=@bene"          # only the new tag
mvn test                                          # all tests (@all tag)
mvn test "-Dcucumber.filter.tags=@smoke"          # smoke set
mvn test -Dbrowser=firefox -Dheadless=false       # different browser
On Windows PowerShell, always quote the -D argument ("-Dcucumber.filter.tags=@smoke") — otherwise PowerShell eats the @.
6

Watch what happens

  • A real Chrome window opens (because browser.headless=false in config-qa.properties).
  • You see your scenario steps run live, slowed to ~500ms each (browser.slow.mo=500).
  • Console prints STARTING SCENARIO / FINISHED SCENARIO from Hooks.
  • On failure: a screenshot is captured automatically and attached to the report.
7

See the result (Allure report)

mvn allure:serve

Opens a browser with a full visual report — pass/fail counts, scenario timeline, screenshots on failure, all the @Step("...") labels from your Page Object methods.

Bene analogy: The Allure report is your "audit trail". Every action a tester takes (click, type, validate) is logged like a transaction journal — perfect for showing your boss / pasting into a JIRA ticket.

4. Where to look when things break

SymptomFirst place to look
Compile failsConsole output of mvn compile — the file:line is in red
Cucumber says "Undefined step"You wrote a Gherkin line but no @Given/@When/@Then matches it in stepdefinitions/. Cucumber prints a template you can paste.
Locator not foundThe page changed, or you're navigating before it loaded. Add waitForElement(selector) in your action.
Browser doesn't openCheck browser.headless in config — also check Playwright browsers are installed (mvn exec:java -e -D exec.mainClass=com.microsoft.playwright.CLI -D exec.args="install").
Test "passes" but I doubt itOpen Allure report — every step shows what was clicked + screenshots.
Got the wrong envYou're on QA by default. Override with -Denv=dev.

5. Mini-glossary (when terms confuse you)

TermPlain meaning
GherkinThe English-like syntax in .feature files (Given/When/Then). Anyone — even a non-coder PM — can read it.
CucumberThe tool that reads Gherkin and runs matching Java code.
GlueThe package path Cucumber scans to find your @Given/@When/@Then Java methods. For us: com.upskill.stepdefinitions + com.upskill.hooks.
Page Object (POM)A Java class that represents one screen of the app. Holds the locators + the actions you can do there.
HooksCode that runs around every scenario — open browser before, close after.
RunnerThe Java class with @RunWith(Cucumber.class) that Maven actually executes.
TagA label like @smoke at the top of a scenario / feature. Used to pick which tests to run.
AllureThe reporting tool. Generates a pretty HTML dashboard from the test run.
ThreadLocalA storage box that gives each parallel thread its own copy — so two tests running side-by-side don't share the same browser.

6. Final mental model (memorise this)

Feature file says WHAT to test (in English).
Step Def says HOW to test it (calls the Page Object).
Page Object says WHERE on the page to do it (locators + actions).
BasePage says HOW THE BROWSER DOES IT (click/type/wait — the Playwright wrappers).
Hooks + Driver say WHEN TO START AND STOP the browser.
Runner says GO. Maven calls it. Allure tells the story.
